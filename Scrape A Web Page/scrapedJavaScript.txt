<span class="comment-copy">@nehemiah: Python is strongly typed (just not statically typed). <code>None</code> is never coerced to <i>anything</i>. Having <code>None</code> silently behave like other types hides errors; that's the <i>opposite</i> of "harmless". If you need a falsy placeholder for the empty sequence, you can use <code>()</code> or <code>''</code>/<code>""</code>, both of which are singletons and can be loaded as cheaply as <code>None</code>. If you want to opt-in to silently treating anything falsy as an empty sequence, you could do <code>for row in data or ():</code>, but no one does that, because passing <code>None</code> to a function expecting a sequence is an error that shouldn't pass silently.</span>

<h2>Explanation of error: 'NoneType' object is not iterable</h2>
<p>In python2, NoneType is the type of None.  In Python3 NoneType is the class of None, for example:</p>
<pre><code>&gt;&gt;&gt; print(type(None))     #Python2
&lt;type 'NoneType'&gt;         #In Python2 the type of None is the 'NoneType' type.

&gt;&gt;&gt; print(type(None))     #Python3
&lt;class 'NoneType'&gt;        #In Python3, the type of None is the 'NoneType' class.
</code></pre>
<h2>Iterating over a variable that has value None fails:</h2>
<pre><code>for a in None:
    print("k")     #TypeError: 'NoneType' object is not iterable
</code></pre>
<h2>Python methods return NoneType if they don't return a value:</h2>
<pre><code>def foo():
    print("k")
a, b = foo()      #TypeError: 'NoneType' object is not iterable
</code></pre>
<h2>You need to check your looping constructs for NoneType like this:</h2>
<pre><code>a = None 
print(a is None)              #prints True
print(a is not None)          #prints False
print(a == None)              #prints True
print(a != None)              #prints False
print(isinstance(a, object))  #prints True
print(isinstance(a, str))     #prints False
</code></pre>
<p>Guido says only use <code>is</code> to check for <code>None</code> because <code>is</code> is more robust to identity checking.  Don't use equality operations because those can spit bubble-up implementationitis of their own.  <a href="https://www.python.org/dev/peps/pep-0008/#programming-recommendations" rel="noreferrer">Python's Coding Style Guidelines - PEP-008</a></p>
<h2>NoneTypes are Sneaky, and can sneak in from lambdas:</h2>
<pre><code>import sys
b = lambda x : sys.stdout.write("k") 
for a in b(10): 
    pass            #TypeError: 'NoneType' object is not iterable 
</code></pre>
<h2>NoneType is not a valid keyword:</h2>
<pre><code>a = NoneType     #NameError: name 'NoneType' is not defined
</code></pre>
<h2>Concatenation of <code>None</code> and a string:</h2>
<pre><code>bar = "something"
foo = None
print foo + bar    #TypeError: cannot concatenate 'str' and 'NoneType' objects
</code></pre>
<h2>What's going on here?</h2>
<p>Python's interpreter converted your code to pyc bytecode.  The Python virtual machine processed the bytecode, it encountered a looping construct which said iterate over a variable containing None.  The operation was performed by invoking the <code>__iter__</code> method on the None. </p>
<p>None has no <code>__iter__</code> method defined, so Python's virtual machine tells you what it sees: that NoneType has no <code>__iter__</code> method. </p>
<p>This is why <a href="https://en.wikipedia.org/wiki/Duck_typing#In_Python" rel="noreferrer">Python's duck-typing</a> ideology is considered bad.  The programmer does something completely reasonable with a variable and at runtime it gets contaminated by None, the python virtual machine attempts to soldier on, and pukes up a bunch of unrelated nonsense all over the carpet.  </p>
<p>Java or C++ doesn't have these problems because such a program wouldn't be allowed to compile since you haven't defined what to do when None occurs.  Python gives the programmer lots of rope to hang himself by allowing you to do lots of things that should cannot be expected to work under exceptional circumstances.  Python is a yes-man, saying yes-sir when it out to be stopping you from harming yourself, like Java and C++ does.</p>

<div class="answercell post-layout--right">
<div class="s-prose js-post-body" itemprop="text">
<p>Code: <code>for row in data:</code><br/>
Error message: <code>TypeError: 'NoneType' object is not iterable</code></p>
<p>Which object is it complaining about? Choice of two, <code>row</code> and <code>data</code>.
In <code>for row in data</code>, which needs to be iterable? Only <code>data</code>.</p>
<p>What's the problem with <code>data</code>? Its type is <code>NoneType</code>. Only <code>None</code> has type <code>NoneType</code>. So <code>data is None</code>.</p>
<p>You can verify this in an IDE, or by inserting e.g. <code>print "data is", repr(data)</code> before the <code>for</code> statement, and re-running.</p>
<p><em>Think about what you need to do next:</em> 
How should "no data" be represented? Do we write an empty file? Do we raise an exception or log a warning or keep silent?</p>

<div class="s-prose js-post-body" itemprop="text">
<p>Another thing that can produce this error is when you are setting something equal to the return from a function, but forgot to actually return anything.</p>
<p>Example:</p>
<pre><code>def foo(dict_of_dicts):
    for key, row in dict_of_dicts.items():
        for key, inner_row in row.items():
            Do SomeThing
    #Whoops, forgot to return all my stuff

return1, return2, return3 = foo(dict_of_dicts)
</code></pre>
<p>This is a little bit of a tough error to spot because the error can also be produced if the row variable happens to be None on one of the iterations. The way to spot it is that the trace fails on the last line and not inside the function.</p>
<p>If your only returning one variable from a function, I am not sure if the error would be produced... I suspect error "'NoneType' object is not iterable in Python" in this case is actually implying "Hey, I'm trying to iterate over the return values to assign them to these three variables in order but I'm only getting None to iterate over"</p>

<blockquote>
<p>pola.py</p>
</blockquote>
<pre><code>@app.route("/search")
def search():
    title='search'
    search_name =  request.form.get('search')
    
    search_item =   User.query.filter_by(id=search_name).first()

    return render_template('search.html', title=title, search_item=search_item  ) 
</code></pre>
<blockquote>
<p>search.html (wrong)</p>
</blockquote>
<pre><code>{% for p in search %}
{{ p }}
</code></pre>
<blockquote>
<blockquote>
<p>search.html (correct)</p>

<pre><code>&lt;td&gt;{{ search_item  }}&lt;/td&gt;

<div class="s-prose js-post-body" itemprop="text">
<p>It means <code>data</code> is <code>None</code>, which is not an iterable. Adding an <code>or []</code>* prevents the exception and doesn't print anything:</p>
<pre><code>for row in data or []:  # no more TypeError!
    print(row)
</code></pre>
<p>* <em>credits to some earlier comments; please beware that raising an exception may be a desired behavior too and/or an indicator of improper <code>data</code> setting.</em></p>
